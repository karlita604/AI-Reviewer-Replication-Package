# Replication Package

### When the Story Changes the Review: Measuring Framing Robustness in AI Code Reviewers

This package contains the dataset, review harness, analysis code, and processed
results for the study. The paper holds the task and code diff of a pull request
fixed and varies only the pull-request *description*, then measures whether eight
open-weight AI code reviewers and one closed reviewer change their approval of
known-bad code. This README explains what is included, how it is organized, and
how to reproduce the results.

This package is anonymized for double-anonymous review: it contains no author
names, affiliations, or identifying links.

## What is included, and what is not

To keep the package small, it ships the **processed** results that the paper's
numbers, tables, and figures are computed from — the parsed review verdicts
(`verdicts.csv`), the bug-detection labels, the signal-detection tables, the
bug-question probe scores, and the regression outputs. The **raw per-trial JSON**
(roughly 200 MB of full model responses for ~38,000 reviews) is **not** included;
it can be regenerated by running the harness (see *Reproducing the results*). The
prompt-ordering ablation, which is not part of this paper, is also omitted.

## The dataset

Each patch folder under `data/` contains the same task shown under four
description conditions — *hedged* (the tentative baseline), *confident*,
*confident-plus-neutral*, and *unsupported-claims* — together with the code diff
and machine-readable bug metadata. Known-bad patches carry one documented,
reviewer-visible defect and a documented triggering condition in their metadata;
verified-good patches carry a written argument for correctness. `tracker.csv` and
the audit files record the two-layer validation described in
`protocols/PATCH-VALIDATION.md`.

## Code

The harness and analysis live in `src/`:

- `reviewer.py`, `run_experiment.py` — run the open-weight reviewers locally and
  collect structured verdicts.
- `api_reviewer.py` — run the closed reviewer (Claude Sonnet) through its vendor
  API.
- `patches.py` — load patches and build the four description conditions.
- `judge_detection.py` — label each review CAUGHT / MISSED against the planted bug.
- `regression.py` — the GEE logistic regression and odds ratios.
- `sdt.py` — the signal-detection decomposition (sensitivity and approval bar).
- `bug_question_analysis.py` — the verdict-free bug-question probe.
- `mitigation_analysis.py` — the RQ4 scorecard.
- `analyze.py`, `figures.py` — aggregate results and render figures.
- `validate.py`, `make_validation_sample.py` — patch validation and sampling.

## Reproducing the results

### Step 1 — Environment

Python ≥ 3.10. Install `torch` first with the CUDA build for your machine, then
the remaining dependencies:

```bash
pip install torch --index-url https://download.pytorch.org/whl/cu124
pip install -r requirements.txt
```

The open-weight reviewers were run on a single NVIDIA H100 (80 GB) GPU each;
the two 70B models were loaded in 4-bit and the six smaller models in 16-bit.
The closed reviewer needs `ANTHROPIC_API_KEY` in the environment.

### Step 2 — Recompute results from the included verdicts (no GPU needed)

The included `verdicts.csv` files are the primary outputs. Run the analysis
scripts from the package root in this order:

```bash
# Detection labels (CAUGHT / MISSED) — must run before analyze.py uses them
python src/judge_detection.py          # writes results/detection/detection.csv

# Main analysis (approval rates, change rates, McNemar, qualitative flips)
python src/analyze.py results/verdicts.csv

# GEE logistic regression and odds ratios
python src/regression.py

# Signal-detection decomposition (d', beta)
python src/sdt.py

# Bug-question probe (verdict-free sensitivity check)
python src/bug_question_analysis.py

# RQ4 mitigation scorecard and lift table
python src/mitigation_analysis.py

# Figures (approval_by_condition.png, change_rate.png)
python src/figures.py
```

All outputs land in `results/` and overwrite the included files.

### Step 3 — Regenerate raw reviews (optional, GPU required)

To recollect the full ~200 MB of per-trial JSON responses from scratch:

```bash
# Size the job first
python run_experiment.py mode=count_trials

# Run per-patch (locally or via slurm/ — see slurm/run_array.sbatch)
sbatch slurm/run_array.sbatch                        # default: llama3_70b
sbatch slurm/run_array.sbatch model=qwen2_5_72b

# After all array tasks finish, consolidate into verdicts.csv
sbatch --dependency=afterok:<JOB_ID> slurm/analyze.sbatch
```

For the frontier (API) reviewer, see `protocols/FRONTIER-PLAN.md`.
Configurations for each experiment variant are in `conf/`.

### Where each result comes from

- **RQ1 (framing and false approvals):** `results/verdicts.csv`,
  `results/analysis/approval_rates.csv`, `results/analysis/regression_odds_ratios.csv`,
  `results/analysis/rq1_change_rates.csv`, `results/analysis/qualitative_flips.txt`.
- **RQ2 (lowered bar, not lost sensitivity):** `results/detection/detection.csv`,
  `results/analysis/sdt/`, `results/phase8_good/analysis/sdt/`, and the probe
  scores under `results/bug_question/` and `results/good_bug_question/`.
- **RQ3 (scale and the closed reviewer):** `results/phase8/`, `results/phase8_good/`,
  and `results/frontier/`.
- **RQ4 (prompt-level fixes):** `results/mitigation/analysis/scorecard.csv` and
  `results/mitigation/analysis/lift_table.csv`.
- **Patch validation:** `results/validation/` and `protocols/PATCH-VALIDATION.md`.

## Protocols

`protocols/` holds the methodology documents:

- `PROTOCOL.md` — review protocol, framing conditions, experimental design.
- `PATCH-VALIDATION.md` — two-layer patch validation (automated + manual audit).
- `JUDGE-PROTOCOL.md` — bug-detection judge rubric and validation plan.
- `MITIGATION-PLAN.md` — pre-registered RQ4 mitigation experiment design.
- `FRONTIER-PLAN.md` — design decisions for the closed-model (Claude Sonnet) extension.
